# bench/eval -- Nexus model-quality eval harness

The reusable eval harness the program design (`NEXUS_PATH_TO_OUTPERFORM_v0.1.md`)
calls for, and the real home of the zero-byte `scripts/benchmark_inference.py`.
This first slice implements **task family 1 (Nexus.Builder workflow synthesis)** and
the **Track C MVE**: verifier-guided best-of-N on a 50-example held-out Builder set,
against the unchanged Qwen3-30B-A3B base model.

It is a harness, not a one-shot script: the cortex client, the prompt template, the
verifier oracle, the dataset, and the runner are separate, importable pieces. Adding
task families 2-6 means adding a dataset + a scorer and reusing everything else.

## The single most important signal

`valid@1` and `valid@k` on Builder, because task family 1 has a **free programmatic
verifier** -- no human or LLM judge, so the metric is not arguable. A candidate
workflow is `valid` only if it clears all three stages:

```
JSON parse  ->  node whitelist  ->  n8n validate_workflow
```

Stages 1-2 are a faithful port of Nexus.Builder NODE 4 ("Parse and Validate").
Stage 3 is the real n8n-mcp `validate_workflow` tool (the `WorkflowValidator` class,
backed by the 806-node SQLite DB), reached through `oracle_bridge/validate_oracle.js`.
No live n8n instance is required -- this is the free oracle the design leads with.

## Metrics

All three are reported with a **bootstrap 95% CI** resampled over the 50 specs
(`bench.stats.bootstrap_ci`). A lift is only real when the CIs do not overlap.

| Metric | What it is | Role |
|---|---|---|
| `valid@1` (greedy) | temperature 0, one shot, fraction that validate | **the baseline bar** |
| `valid@1` (sampled) | expected pass rate of a single temp=0.8 draw | isolates the sampling effect from greedy-vs-sampled |
| `valid@N` (best-of-N) | a spec is solved if ANY of N samples validate | **the Track-C-raised bar** Tracks A/B must beat |

`valid@N` (best-of-N) equals verifier-picked best-of-N for a pass/fail oracle: the
verifier returns the first candidate that validates, so "any of N pass" is exactly
"best-of-N succeeds". `best_of_k` for every k in 1..N is also reported, so the
marginal value of each extra sample is visible (the inference-cost-vs-quality curve).

The **cost axis** is recorded alongside quality: completion tokens (greedy vs
best-of-N), the inference multiplier best-of-N buys, and p50/p95 latency.

## The held-out set

`datasets/builder_heldout_v1.jsonl` -- 50 Builder specs (`{id, domain, operation,
description, required_inputs, expected_outputs, source}`), regenerated by
`build_dataset.py`. Honest provenance: the episodic transcript log was ~empty at
build time (no logged Builder runs to mine), so per the design's allowance the set
is 30 specs lifted from the real `automation/workflow-registry.json` (the actual
workflows Nexus runs) plus 20 synthesized in-distribution specs. "Held out" =
held out from any future Track A/B training; no model has been fine-tuned, so the
whole set is unseen for Track C by construction. The JSONL is frozen and committed;
re-running the generator produces byte-identical output.

## Two prompt conditions

The runner takes `--prompt-variant`:

- **`faithful`** (default, the committed `baseline_v1.json`): the frozen production
  Nexus.Builder GENERATE prompt, verbatim. This is the true bar.
- **`augmented`** (`baseline_v1_promptfix.json`): the same prompt plus rules that
  close the systematic, deploy-time-fixable gaps the base model shows under
  `validate_workflow` (nodes emitted without a unique `id`, empty Code bodies). This
  is a logged prompt experiment -- the design explicitly treats prompt changes as
  separate experiments -- and answers "what can test-time compute reach once the
  trivial blockers are removed?"

## Run it

```bash
# baseline + best-of-8, faithful production prompt -> baseline_v1.json
python -m bench.eval.run_builder_eval

# the prompt-fix variant
python -m bench.eval.run_builder_eval --prompt-variant augmented \
    --baseline-out bench/eval/baseline_v1_promptfix.json --out-dir bench/eval/results/augmented

# smoke test on 2 specs
python -m bench.eval.run_builder_eval --limit 2 --out-dir bench/eval/results/_smoke

# regenerate the dataset
python -m bench.eval.build_dataset
```

Prereqs: the vLLM cortex live on `:8000` (auto-detected) and Node available with the
`Nexus_N8N-MCP` checkout built (`dist/` + `data/nodes.db`). The oracle uses the
n8n-mcp already vendored at `Nexus_N8N-MCP/`.

## Files

| File | Role |
|---|---|
| `cortex_client.py` | stdlib OpenAI-compatible client for vLLM `:8000`, n-sampling + seed |
| `builder_prompt.py` | frozen Builder GENERATE prompt (faithful + augmented variants) |
| `verifier.py` | parse -> whitelist -> validate oracle; long-lived Node bridge |
| `oracle_bridge/validate_oracle.js` | faithful bridge to n8n-mcp `validate_workflow` |
| `build_dataset.py` | reproducible generator for the frozen held-out set |
| `run_builder_eval.py` | the runner: emits `runs.yaml`, results JSON, `baseline_v1.json` |
| `baseline_v1.json` | **the committed bar** -- every later claim is "beats baseline_v1, CIs disjoint" |

## Outputs

- `baseline_v1.json` -- the pinned bar (scores + CIs + full config + hardware).
- `results/<variant>/runs.yaml` -- the run config (model id, quant, sampling, seed,
  verifier profile, harness git SHA, vLLM version) per the design's `runs.yaml` rule.
- `results/<variant>/builder_best_of_n_results.json` -- full per-spec detail.

## Result (v1 baseline, 2026-06-16)

Run against the live Cortex (`Qwen3-30B-A3B-Instruct-2507-AWQ-4bit`, vLLM 0.15.1,
`runtime` profile, seed 0, N=8):

| Condition | valid@1 (greedy) | valid@1 (sampled) | valid@8 (best-of-8) | inference cost |
|---|---|---|---|---|
| `faithful` (production prompt) | **0.00** | 0.00 | **0.00** | 8.8x tokens |
| `augmented` (gap-closing prompt) | 0.00 | 0.00 | **0.00** | 7.4x tokens |

All CIs are `[0.0, 0.0]` -- every one of the 50 specs fails. **Test-time compute
gives zero lift here**, because the failures are systematic n8n-schema errors, not
random noise: all 8 samples repeat the same class of mistakes, so best-of-N has
nothing valid to pick. Best-of-8 is therefore strictly negative EV on this task --
~8.8x the completion tokens (335k vs 38k) and ~3x p95 latency for no quality gain.

The oracle is not rigged to always fail: a hand-built minimal workflow validates
`true` through the same bridge. The base model simply does not produce schema-valid
n8n JSON. Dominant residual failure classes (from `validate_error_histogram`):

1. `Node position must be an array with exactly 2 numbers [x, y]` -- near-universal
2. `Code cannot be empty` -- Code nodes emitted without a `jsCode` body
3. `Connection uses node ID '...' instead of node name` / `Connection to non-existent
   node` -- the model wires connections by id, but n8n keys connections on node *name*
4. `Invalid value for 'operation'/'resource'/'authentication'` -- per-node API enums

**So what.** This is the harness doing its job: it pins the Builder bar at 0 and
says *why*. The next experiment is NOT more sampling. It is, in cost order: (a) a
deterministic post-processor for the mechanical errors (position, connection-by-name)
+ a 1-2 shot prompt seeded with a real valid registry workflow; if that still can't
clear it, (b) the **Track B hot-path LoRA adapter** trained on valid workflow JSON --
and this 0.0 baseline is the quantitative case for spending those GPU-hours.

## Scope and honest limits

- One seed (0) by default. The design's fuller protocol (seeds {0,1,2}) is supported
  by re-running with `--seed`; the bootstrap-over-specs CI is the more important
  interval for a 50-example set and is always reported.
- The oracle is `validate_workflow` (static structural + node validation), not live
  execution. Exec-success (the design's ultimate Builder signal) needs a live n8n and
  is out of scope for this MVE.
- No gold-JSON comparison: the verifier is the ground truth, which is the point of
  leading with a verifiable task.
